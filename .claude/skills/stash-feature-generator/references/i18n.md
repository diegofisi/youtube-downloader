# i18n (es/en) — minimalist, no dictionaries

Every text carries its pair inline. There are no translation files or keys. This system is
for UI-facing strings only and stays as-is (code comments are English; see conventions.md).

## Text generated from TS: `t(es, en)`

```ts
import { t } from '../../core/i18n';
showToast(t('Añadido a la cola', 'Added to queue'), it.title, 'done');
btn.innerHTML = `${I.spinner} ${t('Analizando…', 'Analyzing…')}`;
```
`t()` evaluates `getLang()` AT CALL TIME. Spanish first, English second, always.

## Static HTML (index.html): `data-en*` attributes

The HTML is written in Spanish; `applyStaticI18n()` (first call in main.ts) translates it when
`getLang() === 'en'`:

| Attribute | Replaces | Real example |
|---|---|---|
| `data-en` | The element's text (text nodes only if it has children: doesn't destroy SVGs) | `<button id="btn-analyze" data-en="Preview videos">Previsualizar</button>` |
| `data-en-ph` | The `placeholder` | `<input id="search-input" placeholder="Buscar en YouTube…" data-en-ph="Search YouTube…">` |
| `data-en-title` | The `title` (tooltip) | `<button title="Minimizar" data-en-title="Minimize">` |

Every new index.html element with visible text MUST carry its `data-en*`.

## Records/constants with text: getters (lazy evaluation)

A module-level `Record` is evaluated at import time, BEFORE the language is known… unless you
use getters. Mandatory pattern (STATUS_META, QMAP, Q_LABEL):

```ts
export const STATUS_META: Record<Status, { readonly label: string; … }> = {
  ok: {
    get label() { return t('Descargable', 'Downloadable'); },   // ✅ evaluated on read
    color: 'var(--success)',
  },
};
// ❌ NEVER: ok: { label: t('Descargable', 'Downloadable') }  ← frozen in one language
```
Alternative for arrays: a lazy function — `const MODE_DEFS = () => [ … ]` (descargar.ts),
`const getSteps = () => [ … ]` (onboarding.ts, "so we don't depend on i18n's import order").

## `setLang` reloads the app

`setLang(l)` persists to localStorage (`stash.lang`) and calls `location.reload()`. That's why
NOTHING needs re-rendering on language change: the Settings selector simply calls setLang.
Don't build "hot re-translation" mechanisms.

## What NOT to translate

- Formats and technical values: `MP4`, `MKV`, `WebM`, `MP3`, `M4A`, `Opus`, `4K`, `1080p`, bitrates.
- Product/brand names: Stash, YouTube, yt-dlp, ffmpeg, deno.
- yt-dlp templates (`%(title)s [%(id)s]`), URLs, paths, ids, localStorage keys.
- BACKEND error messages: Spanish only (Result<_, String>, user-facing — until backend i18n
  exists); the frontend shows them verbatim.
- Texts identical in both languages are written twice: `t('Video + audio', 'Video + audio')`.

## Dates and locales: `getLang()`

For language-dependent formatting, pass the lang as the locale (shared/lib/format.ts):

```ts
new Date(secs * 1000).toLocaleString(getLang(), { day: '2-digit', month: 'short', … });
```
`timeAgo` uses `t()` for its units ("hace 5 min" / "5 min ago"). In tests, without localStorage
the default is `'es'`: expected strings are the Spanish ones (format.test.ts).
